# Overview

## Что это за проект

Video Lecture RAG Assistant — это локальная система для вопросов по видеолекциям.

Пользователь задаёт вопрос по содержанию лекции. Система ищет релевантные фрагменты в транскрибированном материале и формирует ответ только на основе найденного контекста. Первая версия делается как прагматичный MVP: без лишней инфраструктуры, с минимальным числом moving parts и с расчётом на запуск на одном ноутбуке.

Система нужна, чтобы быстро находить ответы по длинным лекциям без ручного просмотра видео. Для MVP важнее надёжный базовый путь «загрузить лекцию → задать вопрос → получить grounded-ответ», чем богатый интерфейс, автоматизация всех этапов или максимальная точность навигации по таймкодам.

---

## Что входит в MVP

В первую версию входит:

- локальный запуск на ноутбуке;
- ingest (загрузка) локального видеофайла;
- транскрибация лекции;
- чтение `transcript.json` в C#;
- chunking в C#;
- построение векторов;
- индексация в Qdrant;
- ответ на вопрос по лекциям;
- возврат источника:
  - название лекции;
  - номер чанка;
  - примерное место в лекции.

Практически это означает два рабочих сценария:
1. Администратор вручную подготавливает лекцию к поиску.
2. Пользователь задаёт вопрос и получает ответ по содержанию лекции.

---

## Что не входит в MVP

Сознательно не делаем:

- downloader по ссылкам;
- микросервисы;
- очереди и брокеры сообщений;
- внутренний REST между C# и Python;
- word-level timestamps;
- web UI;
- историю чатов;
- auth и роли пользователей;
- production deployment;
- автоматический reindex.

Эти ограничения приняты не потому, что они не нужны вообще, а потому что они не нужны для первого рабочего результата.

---

## Основной пользовательский и технический сценарий

### Пользовательский сценарий

1. Администратор запускает ingest (загрузку) локального видео.
2. Система транскрибирует лекцию, режет её на чанки и индексирует векторы.
3. Пользователь задаёт вопрос через API.
4. Система находит релевантные чанки, вызывает модель ответа и возвращает результат.

### Технический сценарий

- Python helper получает локальный файл, делает транскрибацию и пишет `transcript.json`.
- C# читает `transcript.json`, выполняет chunking, строит embeddings и пишет данные в Qdrant.
- При вопросе C# строит embedding вопроса, ищет top-k в Qdrant и формирует ответ через генератор ответа.

Главное правило: Python не участвует ни в chunking, ни в retrieval, ни в ответе пользователю.

---

## Обязательный интерфейс MVP

Для первой версии обязательны только:

- **CLI ingest**
- **Minimal API ask**

Это канонический путь MVP. Он должен быть реализован в любом случае.

Дополнительно допускается:
- **Telegram bot** — как optional-интерфейс, если остаётся время.

Не входит в MVP:
- **web UI**.

---

## Ограничения, которые должна помнить вся команда

- **C# — центр системы.** Вся оркестрация находится в C#.
- **Python helper только транскрибирует и пишет `transcript.json`.**
- Qdrant хранит только retrieval-данные (данные для ответа на вопрос пользователя).
- Если меняется provider/model embeddings, требуется rebuild.
- Telegram bot не должен влиять на основную архитектуру.
- Любое усложнение, не улучшающее основной путь MVP, должно считаться подозрительным.
- Сначала команда доводит до рабочего состояния один основной локальный стек провайдеров; запасные реализации добавляются только потом.

---

## Что команда должна считать Done для MVP

MVP можно считать готовым, если:

1. Команда может выполнить:
   ```bash
   video_rag ingest lecture.mp4
   ```
2. Minimal API принимает вопрос и возвращает ответ.
3. Ответ либо полезен, либо система честно сообщает, что релевантного контекста нет.
4. В ответе есть ссылка на источник в формате «лекция + примерное место».
5. Вся система запускается локально по инструкции из репозитория.
