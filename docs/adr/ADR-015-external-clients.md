# ADR-015 — Внешние клиенты используют основной HTTP API

- **ID:** ADR-015
- **Статус:** Принято

## Контекст

В проекте есть основной RAG-контур:

```text
ingest -> transcript -> chunks -> embeddings -> Qdrant -> ask -> answer
```

Этот контур должен оставаться в основном C# приложении. При этом для использования системы могут понадобиться разные интерфейсы:

- локальный web-интерфейс;
- Telegram bot;
- другие клиентские приложения.

Если каждый клиент начнёт напрямую работать с Qdrant, Python helper, embeddings или answer provider, RAG-логика быстро начнёт дублироваться и расходиться между приложениями.

## Решение

Клиентские приложения реализуются как внешние thin clients.

Они обращаются к основному приложению через публичный HTTP API:

```text
Client -> HTTP API -> Application services -> RAG pipeline
```

Основной endpoint для вопросов:

```text
POST /ask
```

Health check:

```text
GET /health
```

Общие DTO для API и клиентов могут быть вынесены в shared project, например:

```text
shared/VideoRag.Contracts
```

## Текущие клиенты

В проекте допускаются клиентские приложения в папке:

```text
clients/
```

Примеры:

```text
clients/VideoRag.TelegramBot
clients/VideoRag.WebUi
```

Клиенты считаются отдельным интерфейсным слоем, а не частью RAG-core.

## Правила для клиентов

Клиенты могут:

- принимать пользовательский ввод;
- вызывать HTTP API основного приложения;
- отображать ответ;
- отображать источники;
- проверять доступность API через health check;
- использовать shared contracts.

Клиенты не должны:

- выполнять retrieval;
- строить embeddings;
- генерировать answer;
- обращаться напрямую к Qdrant;
- запускать Python helper;
- читать transcript files;
- читать registry files;
- дублировать application services;
- знать concrete infrastructure implementations.

## Последствия

Плюсы:

- RAG-логика остаётся в одном месте;
- клиенты проще разрабатывать и тестировать;
- Telegram bot, Web UI и будущие интерфейсы используют один контракт;
- основной backend не зависит от конкретного UI;
- добавление нового клиента не требует изменения RAG pipeline.

Минусы:

- для работы клиента должен быть запущен основной API;
- нужно поддерживать совместимость shared contracts;
- изменения внешнего API требуют синхронного обновления клиентов.

## Что это значит для реализации

Разрешено:

- добавлять новые проекты в `clients/`;
- использовать `shared/VideoRag.Contracts`;
- настраивать адрес основного API через конфигурацию клиента;
- реализовывать UI/UX-логику в клиентах.

Запрещено:

- переносить RAG orchestration в клиент;
- давать клиентам прямой доступ к Qdrant;
- вызывать embedding provider или answer provider из клиента;
- запускать транскрибацию из клиента в обход основного приложения;
- читать файлы `data/transcripts` или `data/registry` из клиента.

## Какие варианты были отвергнуты

### Дублировать RAG-логику в каждом клиенте

Отклонено, потому что это приводит к расхождению поведения между CLI, Web UI и Telegram bot.

### Подключать клиентов напрямую к Qdrant

Отклонено, потому что retrieval — часть application logic, а не UI logic.

### Делать отдельный backend для каждого клиента

Отклонено, потому что это усложняет локальный запуск и создаёт несколько orchestration centers.

### Считать Web UI частью RAG-core

Отклонено. Web UI — интерфейсный слой. RAG-core остаётся в основном приложении.
